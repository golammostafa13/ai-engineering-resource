# LLM Post-Training: Aligning and Customizing Language Models

This document details the 10-step progression of **Post-Training**—the sequence of operations used to refine a raw, pre-trained base model into a helpful, conversational, and agent-ready assistant.

---

## 🏛️ Part 1: Instruction Alignment & Imitation (Steps 1–3)

### 1. Pre-trained Base Model (The Autocomplete Engine)
* **Concept:** The starting model has only been trained to predict the next token across massive text datasets.
* **Behavior:** It has no concept of instructions or conversations. If asked: *"How do I format a date in Python?"*, it may autocomplete with other programming questions rather than providing the code.

### 2. Supervised Fine-Tuning (SFT)
* **Concept:** The model is trained on curated conversational pairs containing instructions and corresponding high-quality answers.
* **Behavior:** By training with standard cross-entropy loss on these templates, the model learns to **imitate** the style and formatting of a helpful assistant. It transitions from a text autocompleter to an instruction follower.

### 3. The Imitation Ceiling
* **Concept:** SFT is restricted because it relies purely on imitation.
* **Behavior:** If the SFT dataset contains inaccuracies, biases, or low-quality phrasing, the model will copy them. Additionally, SFT does not teach the model the boundaries of what it *should not* do (safety, refuse execution).

---

## 📊 Part 2: Preference Optimization & Reward Systems (Steps 4–8)

### 4. Preference Data Collection
* **Concept:** Multiple candidate responses are generated by the SFT model for a single query. These are evaluated by human judges or strong LLMs.
* **Behavior:** The outputs are labeled as pairs: **Winner (Preferred)** and **Loser (Dispreferred)**, capturing safety, accuracy, and format preferences.

### 5. Reward Modeling (RLHF Route)
* **Concept:** A separate, smaller neural network (the **Reward Model**) is trained on the preference dataset.
* **Behavior:** The Reward Model learns to input any text sequence and output a scalar score (e.g., +2.5 for helpful and safe text, -3.0 for hazardous or toxic output) matching human preference trends.

### 6. Reinforcement Learning via PPO (Proximal Policy Optimization)
* **Concept:** The SFT model acts as a policy in a reinforcement learning environment, generating answers to earn high scores from the Reward Model.
* **Behavior:** The model's weights are optimized to maximize the reward score. A KL-divergence penalty is added to keep the active model's weights close to the original SFT baseline, preventing it from producing gibberish that tricks the reward function.

### 7. Direct Preference Optimization (DPO)
* **Concept:** DPO simplifies alignment by bypassng the need for a separate Reward Model and complex PPO reinforcement loops.
* **Behavior:** DPO derives a mathematical link showing that the model can be updated directly on the Winner/Loser dataset using a binary cross-entropy loss. It scales the probability of the preferred response and reduces the probability of the dispreferred response relative to the starting model.

### 8. Kahneman-Tversky Optimization (KTO)
* **Concept:** An alternative to pairwise preference optimization (Winner vs. Loser pairs).
* **Behavior:** KTO trains on individual data points labeled as simply binary feedback (Thumbs Up / Thumbs Down). This makes it significantly easier to collect reinforcement data from real-world chat interfaces.

---

## 🛠️ Part 3: Inference & Agentic Execution (Steps 9–10)

### 9. Rejection Sampling (Best-of-N)
* **Concept:** A test-time/inference method to boost response quality without retraining weight parameters.
* **Behavior:** The model generates $N$ independent candidate answers. A separate reward model or verification script evaluates all $N$ options, returning only the single highest-scoring output to the user.

### 10. Relevance for Agent Builders
* **Concept:** Why post-training is indispensable for building active AI agents.
* **Behavior:** SFT and preference tuning are what train the model to:
  * Parse complex JSON schemas and match tool configurations.
  * Stop generation immediately when a tool execution token is emitted.
  * Enforce safety boundaries and refuse malicious instructions (e.g., refusing to execute unauthorized commands or access restricted databases).
